Key Architectural Principle

Both applications are decoupled at the frontend layer but share the same Node.js backend and PostgreSQL store. This means the user never experiences analytics overhead, and the admin never interferes with live shopping traffic.

02

User Side Flow

User Journey
User Opens Browser → Loads E-Commerce Site → Tracking Script Injected → Session ID Generated
Behavior Capture
Page View + Clicks (x,y) + Scroll Depth + Mouse Move → Event Object
Transmission
Event Object → HTTP POST /api/track → Node.js Backend → Stored in PostgreSQL
What the Tracking Script Captures
  • Click Events — x,y pixel coordinates on every click
  • Scroll Depth — % of page scrolled at each interval
  • Page Views — timestamp + page URL on each navigation
  • Mouse Movement — cursor path for hover heatmaps
  • Session ID — UUID generated once per browser session
How Data is Sent
  • Transport — Lightweight HTTP POST requests (non-blocking)
  • Timing — Events fired asynchronously so page feels fast
  • Batching — Mouse move events are debounced/throttled
  • Payload — JSON: {session_id, event_type, x, y, page, timestamp}
  • Endpoint — POST /api/track on the Node.js server

Data Processing Layer

Database Schema — Core Tables
TableKey ColumnsPurpose
raw_eventssession_id, event_type, x, y, page, tsEvery raw user event as captured
sessionssession_id, start_time, end_time, user_agentOne row per user visit
session_featurestotal_clicks, scroll_depth, duration_sec, pages_visitedAggregated per-session features for ML
feedbacksession_id, message, sentiment_scoreUser feedback + NLP output
Sessionization Logic
  • Session Start — triggered on first page load, UUID assigned
  • Session End — detected via inactivity timeout (e.g., 30 min)
  • Grouping — all events with same session_id are linked
  • Duration — last_event_ts − first_event_ts
  • Page sequence — ordered list of pages visited in session
Feature Engineering
  • total_clicks — COUNT of click events per session
  • avg_scroll_depth — mean scroll % across all pages
  • session_duration — seconds from start to last event
  • pages_visited — distinct page count in session
  • max_funnel_stage — deepest funnel page reached (0–3)
  • hover_density — mouse movement event count / page area

Heatmap Generation
  • Click Heatmap — density map of x,y coordinates per page; brighter = more clicks
  • Scroll Heatmap — colour bands showing how far users scroll; reveals fold
  • Cursor Heatmap — mouse movement paths; reveals attention zones
  • Rendering — aggregated coordinate data passed to a canvas or SVG overlay on a page screenshot
Scroll & Cursor Insights
  • Scroll analysis — identifies the "drop-off line" where users stop reading
  • Above-fold engagement — % of users who scroll past 50%, 75%, 100%
  • Cursor dwell time — zones where cursor pauses longest (high interest)
  • Rage clicks — repeated rapid clicks on same element (frustration signal)

Machine Learning Layer

Drop-Off Prediction Pipeline
session_features table → Feature Vector → XGBoost Model → Drop-off Probability (0–1) → Risk Label (High/Med/Low)
Sentiment Analysis Pipeline
User Feedback Text → Text Preprocessing → NLP Model → Sentiment Score → Positive / Neutral / Negative
Drop-off Prediction — XGBoost
  • Input features — total_clicks, scroll_depth, duration, pages_visited, max_funnel_stage
  • Training data — historical sessions labelled as "converted" or "dropped"
  • Model output — probability score (0.0 – 1.0) per session
  • Threshold — score > 0.65 flagged as high drop-off risk
  • Why XGBoost — handles tabular data well, fast inference, explainable
XGBoost Binary Classification Feature Importance
Sentiment Analysis — NLP
  • Input — free-text user feedback submitted on website
  • Preprocessing — tokenization, stopword removal, lowercasing
  • Model — pre-trained transformer or VADER/TextBlob for scoring
  • Output — sentiment label + score; stored in feedback table
  • Aggregation — daily/weekly positive % shown on dashboard

06

Admin Dashboard Flow

How Admin Accesses the System
  • Separate URL — admin dashboard runs on a distinct domain/port from the e-commerce site
  • Authentication — login-protected; only authorised users can access
  • Data source — reads from the same PostgreSQL database via dedicated GET /api/analytics/* endpoints
  • Real-time refresh — dashboard polls or subscribes to updates at regular intervals

The dashboard is organized into five major insight panels:

① Funnel Drop-off Panel
  • Visual funnel chart showing conversion at each stage
  • Percentage drop between Product → Cart → Checkout → Payment
  • Session counts per stage; filterable by date range
  • Highlights the weakest conversion point
② Heatmap Viewer
  • Toggle between Click, Scroll, and Cursor heatmaps
  • Overlaid on a screenshot of the actual page
  • Filter by page (Product, Cart, Checkout, Payment)
  • Colour scale: cold (blue) = low activity → hot (red) = high
③ Session Analytics Panel
  • Average session duration, total sessions, bounce rate
  • Average clicks per session and scroll depth
  • Top pages by visit count and dwell time
  • Time-series charts showing traffic trends
④ Risk Prediction Panel
  • Distribution of High / Medium / Low risk sessions
  • Table of most recent high-risk sessions with feature values
  • Feature importance chart (which signals drive drop-off)
  • Model confidence score displayed per prediction
⑤ Sentiment Insights Panel
  • Sentiment breakdown — % Positive, Neutral, Negative over time
  • Trending keywords — most common words in negative feedback
  • Feedback feed — live list of recent comments with colour-coded sentiment tags
  • Correlation view — cross-reference sentiment with funnel stage where feedback was submitted
07

End-to-End Data Flow Pipeline

Stage 1 · User Interaction

User Browses the E-Commerce Website

User opens the site. A unique Session ID (UUID) is generated in their browser. As they navigate pages (Product → Cart → Checkout → Payment), the injected tracking script silently listens for events.

Stage 2 · Event Capture

Tracking Script Fires Events

Every click, scroll, page view, and mouse movement is captured with its coordinates, timestamp, page name, and session ID, then serialized into a JSON event object.

Stage 3 · Transmission

HTTP POST → Node.js Backend

Events are sent asynchronously to POST /api/track. The backend validates and writes each event to the raw_events table in PostgreSQL in real time.

Stage 4 · Sessionization

Session Aggregation Job

A scheduled backend job (or trigger) groups raw events by session ID, calculates duration, pages visited, total clicks, max scroll depth, and funnel stage reached — writing results to session_features.

Stage 5 · Analytics Computation

Funnel, Heatmap & Scroll Analysis

The analytics engine queries session data to compute funnel conversion rates, aggregates click/cursor coordinates into heatmap matrices, and calculates scroll distribution curves per page.

Stage 6 · Machine Learning Inference

XGBoost Prediction + NLP Scoring

Session feature vectors are passed to the XGBoost model to generate drop-off probability scores. In parallel, user feedback text is processed by the NLP model to output sentiment labels and scores.

Stage 7 · Result Storage

Predictions Written Back to Database

ML outputs (risk scores, sentiment labels) are stored back into PostgreSQL alongside their source sessions, making them available for dashboard queries.

Stage 8 · Admin Consumption

Admin Dashboard Reads & Visualises

The admin logs into the analytics dashboard. The frontend calls GET /api/analytics/* endpoints which return pre-computed analytics, heatmaps, funnel data, risk flags, and sentiment summaries — displayed as interactive charts and tables.

Simplified One-Line Pipeline
User → Tracking Script → Node.js API → PostgreSQL → Sessionization → Analytics Engine → ML Models → DB Results → Admin Dashboard

Benefits of Separation

Keeping the user-facing e-commerce app and the admin analytics dashboard as two separate applications is a deliberate architectural decision with significant practical advantages.

⚡

Performance Isolation

Analytics queries (heavy aggregations, ML inference) run independently and never slow down the shopping experience for real users.

🔐

Security Boundary

Admin dashboard is on a protected route with authentication. Users can never accidentally — or maliciously — access analytics data.

📐

Clean Separation of Concerns

Developers working on the shopping UI don't need to know anything about ML pipelines. Analytics engineers don't need to touch product pages.

📈

Independent Scalability

If traffic spikes, only the user app needs to scale out. The analytics system can scale separately based on data volume and query load.

🔄

Independent Deployment

The admin dashboard can be updated, redeployed, or taken offline for maintenance without affecting live shoppers at all.

🧪

Easier Testing & Debugging

Each app has a clear responsibility. Bugs in the analytics layer don't affect the shopping experience, and vice versa — simplifying root-cause analysis.

Business & Research Perspective
  • User Trust — users interact with a clean, fast shopping site; tracking is invisible and non-intrusive
  • Data Governance — analytics data access is centralised and auditable through a single admin layer
  • Iterative ML — ML models can be retrained and updated without any changes to the user-facing product
  • Academic Clarity — in research context, the two-system design cleanly separates the data collection domain from the data analysis domain
  • Commercial Realism — mirrors real-world systems like Google Analytics, Mixpanel, and Hotjar, which are always separate from the products they analyse
